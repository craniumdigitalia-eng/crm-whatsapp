# Story 1.1: Idempotência de mensagens por external_id

## Objetivo
Deduplicar mensagens recebidas por um identificador único da mensagem (`external_id`), para que reentregas do Make/Evolution não criem mensagens repetidas nem disparem múltiplas respostas do agente.

## Acceptance Criteria
- [x] AC1: `InboundMessage` (`src/whatsapp/evolution.ts`) passa a carregar `externalId: string` extraído de `key.id` do payload da Evolution; `parseWebhook` o preenche.
- [x] AC2: `Message` (`src/types.ts`) ganha `external_id: string | null` e o schema (SQLite e Supabase) recebe a coluna + índice único parcial (`WHERE external_id IS NOT NULL`).
- [x] AC3: `addMessage` passa a aceitar `externalId?` e a inserção usa semântica idempotente (`INSERT OR IGNORE` no SQLite; retorna `boolean` indicando se houve inserção real).
- [ ] AC4: Teste: enviar o mesmo payload de webhook 2x resulta em **1** linha em `messages` e **1** resposta enviada ao lead (segunda chamada é no-op). _(verificação manual ou por QA)_
- [x] AC5: Mensagens sem `external_id` (ex.: `out` geradas internamente) continuam inserindo normalmente (índice é parcial).

## Escopo

**IN:**
- Propagar `external_id` da borda (`parseWebhook`) até `addMessage`. ✅
- Guarda de dedupe no `handler.ts`: se a mensagem já existe, encerra sem reprocessar. ✅
- Ajuste de tipos em `types.ts`. ✅

**OUT:**
- Criação física da coluna no Supabase (fica na Story 2.1 — aqui só o contrato/tipos e o SQLite local).
- Fila de processamento assíncrono.

## Contexto Técnico
God nodes tocados: `types.ts`, `db.ts`, `crm/leads.ts`, `handler.ts`. Gap #1 do schema. A coluna `external_id` vem de `key.id` (Evolution) ou do ID que o Make repassar. Crítico antes de ligar o canal real — ver [[project/architecture]] risco #3.
