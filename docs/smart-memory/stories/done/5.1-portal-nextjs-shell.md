# Story 5.1: Shell do portal em Next.js App Router

## Objetivo
Subir o app Next.js (App Router) na Vercel como entrypoint do front, com layout/navegação base, preservando as funções `/api` da Wave 2 intactas.

## Acceptance Criteria
- [ ] AC1: App Next.js (App Router) criado e deployado na Vercel; rota `/` carrega o shell do portal.
- [ ] AC2: Layout raiz com **sidebar de navegação** + topbar; segmentos de rota placeholder para os 5 módulos (`/crm`, `/leads`, `/bi`, `/agenda`, `/whatsapp`).
- [ ] AC3: As funções de integração `/api/webhook` e `/api/cron/followup` (Wave 2) continuam respondendo — sem regressão (smoke test).
- [ ] AC4: Domínio `src/` (`crm/leads.ts`, `handler.ts`, `agent/`) não é modificado; o front consome os endpoints existentes.
- [ ] AC5: Build e deploy reproduzíveis (`vercel`); navegação entre segmentos funciona (mesmo com telas placeholder).

## Escopo

**IN:**
- Scaffold Next.js App Router, layout/navegação base, deploy.
- Definição da estrutura de rotas dos módulos.

**OUT:**
- Auth/RBAC (Story 5.2).
- Design branded final (Story 5.3 — aqui é layout neutro/placeholder).
- Conteúdo real dos módulos (5.4+).

## Contexto Técnico
Decisão em [[../../decisions/ADR-003-portal-nextjs]]. Supersede a antiga Story 4.1 (dashboard estático). As funções `/api/leads*` podem virar route handlers do App Router sob demanda nas stories de módulo; nesta story só o shell. Não toca god nodes do backend.
