# Story 3.1: Adapter de canal — Evolution para Make

## Objetivo
Abstrair o envio de WhatsApp para que a saída passe pelo webhook do Make (`MAKE_SEND_URL`) em vez de falar direto com a Evolution, mantendo a função `sendText` como ponto único.

## Acceptance Criteria
- [x] AC1: `config.ts` ganha `makeSendUrl` (`MAKE_SEND_URL`) e `makeWebhookSecret` (`MAKE_WEBHOOK_SECRET`). Vars da Evolution mantidas como fallback/legado.
- [x] AC2: `sendText(phone, text)` em `src/whatsapp/evolution.ts` faz `POST {MAKE_SEND_URL}` com body `{ phone, text }` quando `MAKE_SEND_URL` está definido. Fallback: path Evolution original.
- [x] AC3: Contrato de entrada do Make definido: `parseMakeWebhook` exportada — normaliza `{ phone, name?, text, id? }` → `InboundMessage`. `externalId` = `id` (wamid) ou hash sha256 deterministico (janela 1s) quando ausente.
- [x] AC4: Todos os chamadores de `sendText` (`handler.ts`, `followup/scheduler.ts`, `routes/api.ts`) continuam funcionando sem alteração de assinatura ou import.
- [x] AC5: Validação empírica — 27/27 assertions passando (ver Dev Agent Record). Sem framework de teste nesta story (decisão de equipe — backlog).

## Escopo

**IN:**
- Implementar o adapter de saída via Make.
- Definir/normalizar o contrato de entrada do Make.

**OUT:**
- Implementar a rota `/api/webhook` em si (Story 3.3).
- Remover totalmente o código da Evolution (mantém como fallback).

## Contexto Técnico
God node: `config.ts`. Decisão fixada no CLAUDE.md (Make como ponte). `sendText` é usado por 3 módulos — preservar assinatura é essencial. Ver [[project/architecture]] (topologia de produção).

### Decisões de implementação
- **Abordagem:** refatorar `evolution.ts` (não criar arquivo novo) — zero churn em imports dos 3 chamadores (AC4).
- **Gatilho Make:** `if (config.makeSendUrl)` — quando vazio, cai automaticamente no path Evolution (dev local).
- **`external_id` sem wamid:** hash `sha256(phone|text|epoch_segundos)`. Janela de 1s. Solução definitiva: mapear `message.id` no cenário Make como campo `id`.
- **`fromMe`:** `false` hardcoded em `parseMakeWebhook` — o Make só entrega mensagens do lead; nossas saem via `sendText`.
- **`makeWebhookSecret`:** adicionado ao config (sem validação aqui — escopo da Story 3.3).

Respostas aos 5 pontos do gate:
1. **AC4** ✅ — `grep` confirma: `handler.ts`/`scheduler.ts`/`api.ts` mantêm import e chamada `sendText(phone, text)`; assinatura inalterada.
2. **sendText branch Make** ✅ — payload exatamente `{ phone, text }`. D1: branch é gated por `if (config.makeSendUrl)` → impossível entrar com URL vazia (vazio → fallback Evolution). URL inválida → `fetch` rejeita → `throw` ao caller. Sem falha silenciosa. (Nit: prod com `MAKE_SEND_URL` esquecido cai no Evolution silenciosamente — erro confuso, não silencioso.)
3. **parseMakeWebhook** ✅ — hash determinístico (`sha256(phone|text|epoch_s)`); `fromMe:false` seguro (Make só encaminha do lead); payload inválido → `[]`. Falso-positivo de mesmo-segundo documentado.
4. **Fronteira 3.3** ✅ — `parseMakeWebhook` só exportada (sem caller); `makeWebhookSecret` só declarado (sem validação); `webhook.ts` intocado (sem rota `/api/webhook`). Contrato pronto sem invadir a 3.3.
5. `tsc` EXIT=0 ✅.

Observação encaminhada:
- **[CONCERN → Story 3.3]** Dedup por hash é best-effort (só mesmo-segundo): retry do Make >1s depois cai em bucket diferente → não deduplica. Idempotência confiável depende do wamid `id`. A 3.3 deve assumir `id` presente; mapear `message.id` no cenário Make é pendência do usuário (já rastreada). Não-bloqueante para a 3.1 (adapter/contrato cumpre os ACs).
