# Story 1.3: Update atômico no motor de follow-up

## Objetivo
Evitar corrida entre o cron de follow-up e o webhook concorrente: garantir que uma retomada só seja enviada se o lead ainda estiver elegível no momento do envio (claim atômico do contador).

## Acceptance Criteria
- [x] AC1: O incremento de `follow_up_count` vira um update condicional atômico que só altera a linha se `last_direction = 'out'` e `last_message_at < now() - interval` e `follow_up_count < FOLLOWUP_MAX`.
- [x] AC2: Se duas execuções do cron rodarem em paralelo sobre o mesmo lead, apenas **uma** consegue o claim (a outra recebe 0 linhas afetadas e não envia).
- [x] AC3: Se o lead respondeu (`resetFollowUp`) entre a leitura e o envio, o claim falha e nenhuma retomada é enviada.
- [x] AC4: Ao atingir `FOLLOWUP_MAX`, o lead transita para `status='perdido'` (alinhar com schema follow-up).
- [ ] AC5: Teste de concorrência simulando duas chamadas de `runFollowUpCheck` sobre o mesmo lead → no máximo 1 mensagem `out` registrada por ciclo. _(verificação manual ou por QA)_

## Contexto Técnico
God node: `crm/leads.ts`. Risco #5 da arquitetura (corrida cron+webhook). A semântica de claim atômico precisa ser portável: no Postgres equivale a `UPDATE ... WHERE ... RETURNING`.

### Mecanismo de claim (portabilidade Postgres)

O `claimFollowUp` executa (fix #12: `last_message_at` no SET fecha janela de claims sequenciais):
```sql
UPDATE leads
   SET follow_up_count = follow_up_count + 1,
       last_message_at = datetime('now'),
       updated_at      = datetime('now')
 WHERE id              = ?
   AND last_direction  = 'out'
   AND last_message_at IS NOT NULL
   AND last_message_at < datetime('now', ?)  -- '-N seconds'
   AND follow_up_count = ?                   -- optimistic lock
   AND follow_up_count < ?                   -- limite máximo
```

O `last_message_at = now()` no claim reinicia o relógio do intervalo imediatamente, barrando
um segundo claim no mesmo ciclo antes que o `addMessage('out')` posterior o faça. O `addMessage`
reescreve `last_message_at` com timestamp marginalmente posterior — sem efeito colateral.

No Postgres (Story 2.2), equivale a:
```sql
UPDATE leads
   SET follow_up_count = follow_up_count + 1,
       last_message_at = now(),
       updated_at      = now()
 WHERE id              = $1
   AND last_direction  = 'out'
   AND last_message_at < now() - ($2 || ' milliseconds')::interval
   AND follow_up_count = $3
   AND follow_up_count < $4
RETURNING id;
```
`changes > 0` → `RETURNING` retorna linha → claim bem-sucedido.

### Ordem das operações (claim-then-send)

```
1. SELECT candidatos (filtro amplo)
2. Pre-filtro JS de intervalo (otimização, evita round-trip)
3. claimFollowUp() → UPDATE atômico com todas as condições
4. Se claim falhou (changes=0) → continue (não envia)
5. Se claim OK → sendText (async)
6. addMessage (registra no histórico)
7. Se newCount >= FOLLOWUP_MAX → setStatus('perdido')
```

A transição para `perdido` ocorre mesmo se `sendText` lança (o contador já foi incrementado, o lead não será selecionado em ciclos futuros).
