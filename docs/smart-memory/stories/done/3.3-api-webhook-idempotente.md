# Story 3.3: /api/webhook idempotente (Make → agente → resposta)

## Objetivo
Implementar a rota serverless de entrada que recebe `{ phone, name?, text, id? }` do Make (campo `id` = wamid nativo; `external_id` é o nome da coluna no banco — distinção mantida no código), processa de forma idempotente e síncrona (registra → IA → responde via `MAKE_SEND_URL`).

## Acceptance Criteria
- [x] AC1: `POST /api/webhook` aceita o payload do Make e valida campos mínimos (`phone`, `text`). Campos ausentes → 400. Método errado → 405.
- [x] AC2: Dedupe por `external_id` (Story 1.1): payload reentregue não cria mensagem duplicada nem segunda resposta — `handleInbound` retorna sem reprocessar; webhook responde 200 idempotente.
- [x] AC3: O processamento roda **antes** de retornar (síncrono, `await handleInbound`). `maxDuration: 60` em `vercel.json` (ADR-002).
- [x] AC4: A resposta do agente é enviada via adapter Make (Story 3.1 — `sendText` → `MAKE_SEND_URL`); a mensagem `out` é persistida por `handleInbound`.
- [x] AC5: Gate `AUTO_STATUSES` e promoção `novo → em_atendimento` respeitados via `handleInbound` (lógica não duplicada).
- [x] AC6: Validação empírica 25/25 — routing, auth, parse, dedupe path, erro 500, WARN presença/ausência (ver Dev Agent Record).

## Escopo

**OUT:**

### Decisões de implementação
- **Handler thin:** toda lógica de domínio (dedupe, funil, IA, follow-up reset) delegada a `handleInbound`. O webhook só valida auth, parseia e chama.
- **Auth via `x-make-secret` header — fail-closed em produção (hardening pós-QA):**
  - `MAKE_WEBHOOK_SECRET` definido → valida header `x-make-secret` em qualquer ambiente; header errado/ausente → 401.
  - `NODE_ENV === "production"` + secret vazio → 401 imediato + `console.error`. Ingresso de produção (grava no banco + chama Claude) não pode ir ao ar aberto.
  - `NODE_ENV !== "production"` + secret vazio → fail-open intencional para testes locais.
- **Síncrono obrigatório:** `await handleInbound(messages[0])` — não há fire-and-forget. `maxDuration: 60s` em `vercel.json` para cobrir latência do agente Claude.
- **Idempotência:** `handleInbound` chama `addMessage` com `externalId`; se a constraint única (23505) disparar, `addMessage` retorna `false` e `handleInbound` retorna sem responder. O webhook retorna 200 ao Make — sem reprocessamento.
- **WARN de dedupe degradado:** quando `req.body.id` ausente (sem wamid nativo), emite `console.warn("[webhook] WARN: payload sem id nativo — dedupe degradado; mapeie message.id no cenario Make")`. WARN não emitido quando `id` está presente.
- **Erro 500:** capturado pelo `serverError` helper (mesma convenção de todos os handlers em `api/`). O `[api] Error: ...` no log indica falha no processamento (ex: Supabase fora, Anthropic timeout).

### Nota: `tsc --noEmit` e arquivo de teste externo
O `src/followup/scheduler.test.ts` (criado pelo crm-backend na story 3.4) usa Jest sem `@types/jest` instalado — causa erros de typecheck no `tsc --noEmit` padrão. Os erros são exclusivamente desse arquivo; excluindo `*.test.ts`, `src/` e `api/` passam limpos (verificado com tsconfig ad-hoc).
