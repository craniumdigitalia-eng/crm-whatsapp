# Story 2.2: Reescrever db.ts e crm/leads.ts para Supabase

## Objetivo
Trocar a camada de persistência de SQLite para `@supabase/supabase-js` (service_role no servidor) **preservando as assinaturas** das funções do repositório, isolando o resto do código.

## Acceptance Criteria
- [x] AC1: `src/db.ts` passa a exportar um client Supabase criado com `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`; remoção da dependência `node:sqlite`.
- [x] AC2: Todas as funções de `src/crm/leads.ts` mantêm a mesma assinatura pública (`getOrCreateLead`, `getLead`, `findLeadByPhone`, `listLeads`, `getMessages`, `addMessage`, `resetFollowUp`, `updateLeadFields`, `setStatus`, claim de follow-up da 1.3) — viram `async` e retornam `Promise<...>`.
- [x] AC3: Chamadores (`handler.ts`, `agent/agent.ts`, `routes/api.ts`, `followup/scheduler.ts`) passam a `await` as funções; `npm run build` compila.
- [x] AC4: `listLeads` e demais SELECTs usam **colunas explícitas** (não `SELECT *`) — resolve gap #3.
- [x] AC5: `addMessage` usa INSERT + `error.code === '23505'` (idempotência da 1.1 preservada) e atualiza `last_direction`/`last_message_at`.
- [x] AC6: Fluxo end-to-end validado contra o Supabase real — 29/29 assertions ok (2026-06-25). Cobre: `getOrCreateLead` (create + idempotência), `getLead`, `findLeadByPhone`, `listLeads` (colunas explícitas), `addMessage` (insert + dedup `23505`), `getMessages` (ordem `created_at`), `updateLeadFields`, `setStatus`, `resetFollowUp`, `claimFollowUp` (optimistic lock + fix #12).

## Escopo

**IN:**
- Reimplementar acesso a dados no Supabase mantendo contratos.
- Converter funções síncronas em assíncronas e ajustar os pontos de chamada.
- Tratar datas como ISO string (`timestamptz`) — removido `parseSqliteDate`.

**OUT:**
- Migração das rotas para serverless (Wave 2).
- Mudança de canal (Make) — Wave 2.

## Contexto Técnico
God nodes: `db.ts`, `crm/leads.ts` (e ajustes em `handler.ts`). A conversão síncrono→assíncrono é a mudança de maior raio de impacto da migração. Depende de 2.1 (schema), 1.2 (UUID) e 1.3 (claim atômico). Ver [[project/modules]] (contratos a preservar).

## Decisões implementadas

| # | Decisão | Padrão escolhido |
|---|---------|-----------------|
| D1 | `claimFollowUp` | JS client: `.update({...}).eq(...).select('id')` — `data.length > 0` = claim ok. `last_message_at` no SET (fix #12 preservado). |
| D2 | `getOrCreateLead` | SELECT → INSERT → fallback `23505` (não `ignoreDuplicates` que não retorna a linha). |
| D3 | `addMessage` dedup | INSERT + `error.code === '23505'` (índice parcial não suporta `ON CONFLICT (col)` via JS client). |
| D4 | build sem runtime | `config.ts` usa `required()` que emite `warn` mas não lança — build compila mesmo sem `.env`. |
| — | UUID geração | `gen_random_uuid()` default no Postgres; sem `crypto.randomUUID()` no app; sem `id` nos inserts. |
| — | `updated_at` | Trigger Postgres cuida — sem `updated_at` nos SETs. |
| — | Substituição | Substituição total do SQLite — sem dual-driver. |
