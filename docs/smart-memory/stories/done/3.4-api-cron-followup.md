# Story 3.4: /api/cron/followup + Vercel Cron

## Objetivo
Substituir o `node-cron` por uma rota serverless disparada pela Vercel Cron, protegida por `CRON_SECRET`, que executa um ciclo de retomadas usando o claim atômico da Story 1.3.

## Acceptance Criteria
- [x] AC1: `POST/GET /api/cron/followup` exige header/token igual a `CRON_SECRET`; sem ele responde 401.
- [x] AC2: `vercel.json` configura a Vercel Cron chamando a rota no schedule definido (ex.: a cada 5 min ou conforme `FOLLOWUP` config).
- [x] AC3: Um ciclo busca leads elegíveis (índice `idx_leads_followup`), aplica o **claim atômico** (Story 1.3) e envia a retomada via Make (Story 3.1).
- [x] AC4: O `node-cron` e `startFollowUpEngine` são removidos do caminho de produção.
- [x] AC5: Lote limitado/paginado para não estourar o timeout da função (risco #1 da arquitetura); documentar o limite por ciclo.
- [x] AC6: Teste: lead com intervalo vencido recebe exatamente 1 retomada por ciclo; lead que respondeu não recebe; ao atingir `FOLLOWUP_MAX` vira `perdido`.

## Escopo

**IN:**
- Rota cron serverless + config Vercel Cron + auth por `CRON_SECRET`.
- Portar `runFollowUpCheck` (sem o timer em memória).

**OUT:**
- Lógica de claim atômico em si (definida na Story 1.3).
- Mudança das mensagens de rotação (mantém as existentes).

## Contexto Técnico
Resolve risco #1 da arquitetura (cron em serverless). Depende de 1.3 (claim), 2.2 (repositório async), 3.1 (canal Make), 3.2 (scaffold). Atenção ao timeout de função — paginar.

- **(d) GET+POST exigem secret** ✅ — gate de auth após o filtro de método, para ambos.
- **AC4** ✅ — node-cron/startFollowUpEngine dev-only; nenhum handler api/ os importa.
