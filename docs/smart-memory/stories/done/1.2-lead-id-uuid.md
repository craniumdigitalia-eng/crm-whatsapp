# Story 1.2: Migrar Lead.id / Message.id de number para string (UUID)

## Objetivo
Alinhar o modelo de tipos do domínio com o Postgres do Supabase, trocando os identificadores `number` por `string` (UUID), sem quebrar o protótipo.

## Acceptance Criteria
- [x] AC1: Em `src/types.ts`, `Lead.id`, `Message.id` e `Message.lead_id` passam a `string`.
- [x] AC2: Toda assinatura que recebe `leadId: number` em `src/crm/leads.ts`, `src/handler.ts`, `src/agent/agent.ts`, `src/routes/api.ts` e `src/followup/scheduler.ts` passa a `string`.
- [x] AC3: `routes/api.ts` deixa de fazer `parseInt(req.params.id, 10)`; usa o `id` como string (UUID) direto.
- [x] AC4: `getOrCreateLead` retorna lead com `id` string em ambos os backends (SQLite: `randomUUID()` em app; Supabase: `gen_random_uuid()` no DB).
- [x] AC5: `npm run build` compila sem erros de tipo; smoke test do fluxo receber→responder funciona no SQLite (gerando UUID localmente via `crypto.randomUUID()`).

## Escopo

**IN:**
- Alteração de tipos e propagação por todo o grafo de chamadas. ✅
- Geração de UUID no protótipo SQLite (`crypto.randomUUID()`). ✅

**OUT:**
- Reescrita do repositório para Supabase (Story 2.2).
- Mudança de schema de produção (Story 2.1).

## Contexto Técnico
God node central: `types.ts` (importado por 5 módulos). Gap #2 do schema. Mudança propaga para todos os consumidores de `id`. Fazer antes da reescrita de persistência para que o repositório Supabase já nasça tipado em string.
