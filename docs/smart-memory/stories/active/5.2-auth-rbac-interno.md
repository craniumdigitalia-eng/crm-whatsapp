# Story 5.2: Auth interno + RBAC (Supabase Auth SSR + RLS por papel)

## Objetivo
Proteger o portal com login da equipe interna (Supabase Auth via SSR/middleware Next) e controle de acesso por papel, com RLS no Supabase.

## Acceptance Criteria
- [x] AC1: Login via Supabase Auth; middleware Next protege todas as rotas do portal (não autenticado → redirect para login). ✅
- [x] AC2: Modelo de papéis definido (`admin`, `atendente`) persistido (`profiles`, migration 004 + role-lock 006). ✅ **006 confirmado APLICADO** (R4 → `42501`).
- [ ] AC3: RLS habilitado em `leads` e `messages`; funções backend seguem com `service_role`. ❌ migration 005 **NÃO aplicada** — runner provou (sentinela) que anon key LÊ e INSERE em leads/messages.
- [x] AC4: RBAC na UI + backend: ações sensíveis exigem `requireAdmin` (12 endpoints); atendente não edita. ✅ (F1–F12 → 403)
- [~] AC5: Teste negativo (não autenticado + sem papel negados, auth + RLS, e2e). ✅ runner `scripts/test/ac5-negative.mjs` RODADO — N* 11/11, F* 14/14, R4 PASS; R1–R3 (RLS) pendem da 005.

## Escopo

**IN:**
- Supabase Auth SSR + middleware + modelo de papéis + RLS/policies + guardas de UI por papel.

**OUT:**
- Multi-tenant por organização/cliente (planejar como evolução; aqui é single-org com papéis internos).
- Telas dos módulos (apenas os guards).

## Contexto Técnico
Absorve e amplia a [[4.2-auth-supabase-rls]] (de "auth do dashboard" para "auth+RBAC do portal"). Gap #8 do schema (RLS desabilitado). Depende de 5.1 (shell). Coordenar tabela de papéis e policies com a crm-data.

**Matriz de gates:** nenhum endpoint sensível ficou sem gate. Públicos por design têm auth de máquina própria (leadgen HMAC/secret; webhook token Evolution fail-closed; cron Bearer fail-closed; google/callback CSRF state; track/unsubscribe públicos).

**Condição para PASS pleno (gate condicional — depende de ação do usuário):**
