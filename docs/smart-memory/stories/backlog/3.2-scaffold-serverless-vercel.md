# Story 3.2: Scaffold serverless — rotas Express para funções /api

## Objetivo
Reestruturar a aplicação Express de processo longo em funções serverless da Vercel, criando a base (`/api`, `vercel.json`) sobre a qual webhook e cron serão implementados.

## Acceptance Criteria
- [ ] AC1: Estrutura `/api` criada com funções serverless; `src/index.ts` (app.listen) deixa de ser o entrypoint de produção.
- [ ] AC2: As rotas do dashboard (`/api/leads`, `/api/leads/:id`, reply/status/takeover/release/edit) viram funções serverless equivalentes, consumindo o repositório async (Story 2.2).
- [ ] AC3: `vercel.json` configurado (funções + placeholder do cron); `health` endpoint disponível.
- [ ] AC4: Nenhum estado em memória entre invocações (sem `node-cron`, sem singletons mutáveis); client Supabase inicializado por invocação ou reusado de forma stateless-safe.
- [ ] AC5: Deploy de preview na Vercel responde às rotas de leitura (`GET /api/leads`) contra o Supabase.

## Escopo

**IN:**
- Migrar as rotas do `routes/api.ts` para funções serverless.
- `vercel.json`, estrutura de pastas `/api`, variáveis de ambiente na Vercel.

**OUT:**
- `/api/webhook` (Story 3.3) e `/api/cron/followup` (Story 3.4) — só o esqueleto/roteamento aqui.
- Migração para Next.js App Router (decisão de ADR; esta story usa funções `/api` puras).

## Contexto Técnico
Risco #2 da arquitetura: o fire-and-forget do webhook não sobrevive — esta story estabelece o padrão de função stateless. Depende de 2.2 (repositório async). Ver [[project/architecture]].
