# Story 10.1: Multi-tenant de banco compartilhado

> 🟢 **Escala futura — problema bom.** Só executar **quando o volume justificar** (dezenas → centenas de clientes). Gerenciar N Supabase/Vercel cansa nesse volume. Não antecipar: é reforma grande e só compensa com volume. Requer **ADR** dedicado antes de começar.

## User Story
Como **dono do SaaS com centenas de clientes** quero **migrar de "deploy por cliente" para banco compartilhado (org_id em todas as tabelas + RLS + config por org)** para **parar de gerenciar N Supabase/Vercel e escalar a operação de forma sustentável**.

## Objetivo
Reformar a arquitetura multi-tenant: de N deploys isolados para um banco compartilhado com `org_id` em todas as tabelas, RLS por org e configuração por organização, quando o volume tornar o modelo atual insustentável.

## Acceptance Criteria
- [ ] AC1: **ADR de tenancy** aprovado antes de qualquer código, comparando "deploy por cliente" × "banco compartilhado" e definindo o gatilho de volume que justifica a migração.
- [ ] AC2: Todas as tabelas de domínio ganham **`org_id`** (leads, messages, conversations, funnel, finance, demands, groups, meetings, email, subscriptions etc.).
- [ ] AC3: **RLS por `org_id`** garante que nenhum dado de uma org seja visível/alterável por outra; testes de isolamento cobrem os principais caminhos.
- [ ] AC4: **Configuração por org** (integrações, agente, canal, plano) isolada por `org_id` — o que hoje é `integrations_config` por deploy vira por org.
- [ ] AC5: **Estratégia de migração de dados** dos deploys existentes para o banco compartilhado, sem perda e com janela controlada (ou coexistência dos dois modelos durante a transição).
- [ ] AC6: Provisionamento ([[6.2-onboarding-self-service]]), super-admin ([[6.3-painel-super-admin]]), limites ([[9.1-limites-por-plano]]) e backups ([[7.3-observabilidade]]) reescritos/ajustados para o modelo `org_id`.
- [ ] AC7: Performance validada sob carga com múltiplas orgs no mesmo banco (RLS não degrada além do aceitável).

## Escopo

**IN:**
- ADR de tenancy, `org_id` em todo o schema, RLS por org, config por org, migração dos deploys existentes, ajuste dos módulos dependentes de tenancy.

**OUT:**
- Execução antes de haver volume que justifique (é gatilho, não meta imediata).
- Reescrita funcional dos módulos (é reforma de tenancy, não de features).

## Dependências
- Toca praticamente todo o schema — coordenação profunda com crm-data.
- Simplifica [[6.3-painel-super-admin]] e muda [[6.2-onboarding-self-service]] (provisionar org é criar linha, não deploy), [[7.3-observabilidade]] (backup de 1 banco), [[9.1-limites-por-plano]] (contagem por org).

## Notas de risco
- **Maior reforma do roadmap.** God node em nível de schema inteiro. Migração de dados de produção de N clientes é a parte mais arriscada — exige plano de rollback e possivelmente coexistência dos dois modelos.
- **Não antecipar.** O roadmap é explícito: só quando o volume pedir. Fazer cedo demais é custo sem retorno; o onboarding automático ([[6.2-onboarding-self-service]]) pode ser o sinal de que o momento chegou.

## Contexto Técnico
Hoje: multi-tenant **por deploy** (1 Supabase + 1 Vercel por cliente), kit `provisioning/`. RLS já existe parcialmente (migration 005, opcional). A reforma torna a RLS por `org_id` obrigatória e central.
