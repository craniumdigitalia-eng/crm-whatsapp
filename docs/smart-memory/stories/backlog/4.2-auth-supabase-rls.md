# Story 4.2: Autenticação do dashboard + RLS

## Objetivo
Proteger o dashboard com Supabase Auth e habilitar RLS nas tabelas, garantindo que dados de leads não fiquem expostos quando o front consultar o Supabase.

## Acceptance Criteria
- [ ] AC1: Login via Supabase Auth no dashboard; rotas/funções de leitura exigem usuário autenticado.
- [ ] AC2: RLS habilitado em `leads` e `messages` (gap #8 do schema); policies criadas para usuário autenticado.
- [ ] AC3: As funções serverless de backend continuam usando `service_role` (ignoram RLS) — comportamento inalterado.
- [ ] AC4: Teste negativo: requisição não autenticada ao dado de leads é negada (RLS + auth).
- [ ] AC5: Documentar a policy inicial e o caminho para multi-tenant (por conta/equipe).

## Escopo

**IN:**
- Supabase Auth no front + habilitar RLS + policies.

**OUT:**
- Multi-tenant completo por organização (planejar como evolução; policy inicial é single-org autenticado).
- Observabilidade (Story 4.5).

## Contexto Técnico
Gap #8 do schema (RLS desabilitado é risco alto quando o front fala direto com o Supabase). Depende de 4.1 (dashboard no ar). Coordenar policies com a crm-data.
