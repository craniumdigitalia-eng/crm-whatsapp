# Story 3.1: Adapter de canal — Evolution para Make

## Objetivo
Abstrair o envio de WhatsApp para que a saída passe pelo webhook do Make (`MAKE_SEND_URL`) em vez de falar direto com a Evolution, mantendo a função `sendText` como ponto único.

## Acceptance Criteria
- [ ] AC1: `config.ts` ganha `makeSendUrl` (`MAKE_SEND_URL`) e mantém as vars da Evolution só como fallback/legado.
- [ ] AC2: `sendText(phone, text)` em `src/whatsapp/evolution.ts` (ou novo `whatsapp/channel.ts`) faz `POST {MAKE_SEND_URL}` com body `{ phone, text }` quando `MAKE_SEND_URL` está definido.
- [ ] AC3: A entrada deixa de depender do parse da Evolution: definir o contrato `{ phone, name, text, external_id }` que o Make entrega (usado pela Story 3.3).
- [ ] AC4: Todos os chamadores de `sendText` (`handler.ts`, `followup/scheduler.ts`, `routes/api.ts`) continuam funcionando sem alteração de assinatura.
- [ ] AC5: Teste: chamada de `sendText` dispara um POST para a URL do Make com payload correto (mock do fetch).

## Escopo

**IN:**
- Implementar o adapter de saída via Make.
- Definir/normalizar o contrato de entrada do Make.

**OUT:**
- Implementar a rota `/api/webhook` em si (Story 3.3).
- Remover totalmente o código da Evolution (mantém como fallback).

## Contexto Técnico
God node: `config.ts`. Decisão fixada no CLAUDE.md (Make como ponte). `sendText` é usado por 3 módulos — preservar assinatura é essencial. Ver [[project/architecture]] (topologia de produção).
