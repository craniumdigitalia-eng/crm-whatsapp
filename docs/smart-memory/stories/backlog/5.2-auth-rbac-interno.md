# Story 5.2: Auth interno + RBAC (Supabase Auth SSR + RLS por papel)

## Objetivo
Proteger o portal com login da equipe interna (Supabase Auth via SSR/middleware Next) e controle de acesso por papel, com RLS no Supabase.

## Acceptance Criteria
- [ ] AC1: Login via Supabase Auth; middleware Next protege todas as rotas do portal (não autenticado → redirect para login).
- [ ] AC2: Modelo de papéis definido (ex.: `admin`, `atendente`, `viewer`) persistido (tabela de perfis/`user_roles`) — coordenar coluna/tabela com a crm-data.
- [ ] AC3: RLS habilitado em `leads` e `messages` (absorve a antiga Story 4.2); policies por usuário autenticado/papel; funções backend seguem com `service_role`.
- [ ] AC4: RBAC na UI: módulos/ações sensíveis (ex.: conectar WhatsApp, BI de custo) restritos a papéis adequados; `viewer` não edita.
- [ ] AC5: Teste negativo: requisição não autenticada e usuário sem papel são negados (auth + RLS), validado e2e.

## Escopo

**IN:**
- Supabase Auth SSR + middleware + modelo de papéis + RLS/policies + guardas de UI por papel.

**OUT:**
- Multi-tenant por organização/cliente (planejar como evolução; aqui é single-org com papéis internos).
- Telas dos módulos (apenas os guards).

## Contexto Técnico
Absorve e amplia a [[4.2-auth-supabase-rls]] (de "auth do dashboard" para "auth+RBAC do portal"). Gap #8 do schema (RLS desabilitado). Depende de 5.1 (shell). Coordenar tabela de papéis e policies com a crm-data.
