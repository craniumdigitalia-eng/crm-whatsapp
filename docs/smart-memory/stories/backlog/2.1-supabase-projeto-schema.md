# Story 2.1: Criar projeto Supabase e aplicar schema

## Objetivo
Provisionar o Postgres de produção no Supabase com o schema completo, incluindo os ajustes de hardening da Wave 0 (coluna `external_id`, constraints).

## Acceptance Criteria
- [ ] AC1: Projeto Supabase criado; `supabase/schema.sql` aplicado com sucesso (tabelas `leads`, `messages`, ENUM `lead_status`, índices `idx_messages_lead`, `idx_leads_status`, `idx_leads_followup`).
- [ ] AC2: `messages.external_id text` + índice único parcial `idx_messages_external_id ... WHERE external_id IS NOT NULL` presentes (incorpora Story 1.1).
- [ ] AC3: Constraint `chk_followup CHECK (follow_up_count >= 0)` aplicada (gap #7 do schema).
- [ ] AC4: `id` de `leads` e `messages` são `uuid DEFAULT gen_random_uuid()`; FK `messages.lead_id → leads.id ON DELETE CASCADE`.
- [ ] AC5: Trigger `trg_leads_updated_at` ativo; `INSERT` de teste em `leads` popula `created_at`/`updated_at` e `UPDATE` atualiza `updated_at`.
- [ ] AC6: RLS permanece **desabilitado** nesta story (acesso por service_role); habilitação fica na Story 4.2.

## Escopo

**IN:**
- Provisionar projeto + rodar migração.
- Garantir que o schema reflete os gaps resolvidos na Wave 0.

**OUT:**
- Reescrita do código de acesso (Story 2.2).
- RLS / policies (Story 4.2).

## Contexto Técnico
Autoria do schema é da crm-data ([[agents/data-engineer/schema]]); esta story aplica e valida. Depende das definições de `external_id` (1.1) e UUID (1.2) para o schema sair completo.
