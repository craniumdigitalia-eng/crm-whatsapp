# Story 3.3: /api/webhook idempotente (Make → agente → resposta)

## Objetivo
Implementar a rota serverless de entrada que recebe `{ phone, name, text, external_id }` do Make, processa de forma idempotente e síncrona (registra → IA → responde via `MAKE_SEND_URL`).

## Acceptance Criteria
- [ ] AC1: `POST /api/webhook` aceita o payload do Make e valida campos mínimos (`phone`, `text`).
- [ ] AC2: Dedupe por `external_id` (Story 1.1): payload reentregue não cria mensagem duplicada nem segunda resposta — retorna 200 idempotente.
- [ ] AC3: O processamento roda **antes** de retornar (síncrono, dentro do timeout da função) — sem fire-and-forget; a lógica reaproveita `handleInbound`.
- [ ] AC4: A resposta do agente é enviada via adapter Make (Story 3.1); a mensagem `out` é persistida.
- [ ] AC5: Respeita o gate `AUTO_STATUSES` (não responde em `humano`/`proposta`/`fechado`/`perdido`) e promove `novo → em_atendimento`.
- [ ] AC6: Teste end-to-end: 1 mensagem nova → 1 lead, 1 `in`, 1 resposta `out`; mesma mensagem 2x → sem duplicação.

## Escopo

**IN:**
- Rota serverless `/api/webhook` adaptando `handler.ts`.
- Processamento síncrono idempotente.

**OUT:**
- Fila/queue externa (alternativa descartada para o volume atual; registrar como ADR se mudar).
- Cron de follow-up (Story 3.4).

## Contexto Técnico
God node: `handler.ts`. Resolve os riscos #2 (fire-and-forget) e #3 (idempotência) da arquitetura. Depende de 3.1 (canal Make), 3.2 (scaffold), 2.2 (repositório), 1.1 (external_id).
