# Story 2.2: Reescrever db.ts e crm/leads.ts para Supabase

## Objetivo
Trocar a camada de persistência de SQLite para `@supabase/supabase-js` (service_role no servidor) **preservando as assinaturas** das funções do repositório, isolando o resto do código.

## Acceptance Criteria
- [ ] AC1: `src/db.ts` passa a exportar um client Supabase criado com `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`; remoção da dependência `node:sqlite`.
- [ ] AC2: Todas as funções de `src/crm/leads.ts` mantêm a mesma assinatura pública (`getOrCreateLead`, `getLead`, `findLeadByPhone`, `listLeads`, `getMessages`, `addMessage`, `resetFollowUp`, `updateLeadFields`, `setStatus`, claim de follow-up da 1.3) — viram `async` e retornam `Promise<...>`.
- [ ] AC3: Chamadores (`handler.ts`, `agent/agent.ts`, `routes/api.ts`, `followup/scheduler.ts`) passam a `await` as funções; `npm run build` compila.
- [ ] AC4: `listLeads` e demais SELECTs usam **colunas explícitas** (não `SELECT *`) — resolve gap #3.
- [ ] AC5: `addMessage` usa `upsert`/`ON CONFLICT (external_id) DO NOTHING` (idempotência da 1.1) e atualiza `last_direction`/`last_message_at`.
- [ ] AC6: Fluxo end-to-end (criar lead, gravar mensagens, qualificar via tool, follow-up) validado contra o Supabase real.

## Escopo

**IN:**
- Reimplementar acesso a dados no Supabase mantendo contratos.
- Converter funções síncronas em assíncronas e ajustar os pontos de chamada.
- Tratar datas como `timestamptz` (remover `parseSqliteDate`).

**OUT:**
- Migração das rotas para serverless (Wave 2).
- Mudança de canal (Make) — Wave 2.

## Contexto Técnico
God nodes: `db.ts`, `crm/leads.ts` (e ajustes em `handler.ts`). A conversão síncrono→assíncrono é a mudança de maior raio de impacto da migração. Depende de 2.1 (schema), 1.2 (UUID) e 1.3 (claim atômico). Ver [[project/modules]] (contratos a preservar).
