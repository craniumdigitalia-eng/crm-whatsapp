# Story 6.2: Onboarding self-service (fluxo cadastro → pagamento → provisiona → wizard)

> **Divisão de responsabilidade com [[9.3-wizard-setup-in-app]]:** a **6.2 (esta)** é o **fluxo que leva o corretor ATÉ o wizard** — cadastro → pagamento (plano único) → conta provisionada sozinha → **redireciona pro wizard**. A **9.3** é o **wizard em si** (as telas de conectar as integrações). Sem sobreposição: aqui termina quando o corretor cai na porta do wizard; o que ele faz lá dentro é a 9.3.

> ✅ **Plano único (R$997/mês)** por [[6.1-cobranca-assinatura]] (sem seleção de tier no checkout). **Provisionamento orquestrado pelo control-plane** ([[../../decisions/ADR-008-plano-de-controle-central]], schema separado no Supabase atual). **Guard-rail:** a instância da Cranium não é tenant; este fluxo cria **corretores** (tenants do SaaS).

## User Story
Como **novo corretor (plano de saúde)** quero **me cadastrar no site, assinar o plano único e ter minha conta provisionada sozinha, caindo direto no wizard de setup** para **começar a configurar o produto sem depender de instalação manual feita pela Cranium**.

## Objetivo
Automatizar o fluxo site → cadastro → **pagamento do plano único** → **conta provisionada automaticamente** (já com o protocolo de plano de saúde pré-carregado, [[6.5-protocolo-plano-saude]]) → **redireciona pro wizard** ([[9.3-wizard-setup-in-app]]), eliminando o passo manual (kit `provisioning/`) que hoje trava a venda em escala.

## Acceptance Criteria
- [ ] AC1: Fluxo de cadastro público: visitante cria conta (e-mail + senha via Supabase Auth) e assina o **plano único** de [[6.1-cobranca-assinatura]] (não há seleção de tier; o checkout é do plano único R$997/mês, com trial se aplicável).
- [ ] AC2: Após pagamento confirmado (webhook de [[6.1-cobranca-assinatura]]), a conta do corretor é **provisionada automaticamente** sem intervenção humana — o que hoje o kit `provisioning/` (`setup.ts`) faz manualmente vira gatilho automático, orquestrado pelo control-plane ([[../../decisions/ADR-008-plano-de-controle-central]]).
- [ ] AC3: A conta provisionada nasce com schema/config prontos e **já com o protocolo de plano de saúde pré-carregado** ([[6.5-protocolo-plano-saude]]); em seguida o corretor é **redirecionado para o wizard de setup** ([[9.3-wizard-setup-in-app]]). Esta story entrega o corretor na porta do wizard; os passos do wizard são da 9.3.
- [ ] AC4: Provisionamento é **idempotente e observável**: se falhar no meio, é retryável e não deixa conta meio-criada; erros notificam o super-admin ([[6.3-painel-super-admin]]).
- [ ] AC5: Tempo total do "pagou → caiu no wizard" é medido e registrado (meta mensurável: **< 5 min** sem intervenção humana no caminho feliz).

## Escopo

**IN:**
- Cadastro público + checkout do **plano único** (integra com [[6.1-cobranca-assinatura]]).
- Gatilho automático de provisionamento pós-pagamento (evoluir o kit `provisioning/setup.ts` para rodar sob demanda/serverless, orquestrado pelo control-plane), **incluindo aplicar o seed de protocolo de plano de saúde** ([[6.5-protocolo-plano-saude]]).
- **Redirecionamento para o wizard** ([[9.3-wizard-setup-in-app]]).

**OUT:**
- **O wizard em si** (telas de conectar OpenAI/Evolution/Google/Meta + confirmar protocolo) — fica em [[9.3-wizard-setup-in-app]].
- O **conteúdo** do protocolo de plano de saúde (o seed) — [[6.5-protocolo-plano-saude]]; aqui só se dispara a aplicação dele no provisionamento.
- Cobrança/plano em si — [[6.1-cobranca-assinatura]].
- A decisão do canal WhatsApp — [[../../decisions/ADR-006-canal-whatsapp-em-escala]].

## Dependências
- Depende de [[6.1-cobranca-assinatura]] (checkout do plano único + webhook de pagamento) para o gatilho de provisionamento.
- O provisionamento é orquestrado pelo control-plane ([[../../decisions/ADR-008-plano-de-controle-central]], schema `control_plane` no Supabase atual).
- Aplica o seed de [[6.5-protocolo-plano-saude]] e **redireciona para [[9.3-wizard-setup-in-app]]**.
- Relaciona com [[6.3-painel-super-admin]] (visibilidade das contas provisionadas).

## Notas de risco
- **Provisionamento automático no modelo "deploy por cliente"** é o ponto mais complexo: hoje cada cliente é 1 Supabase + 1 Vercel. Provisionar isso sozinho a cada venda pode exigir automação de infra (criar projeto Supabase + deploy Vercel via API) OU antecipar parte da Fase 5 (banco compartilhado `org_id`). O [[../../decisions/ADR-008-plano-de-controle-central]] define quem orquestra isso (o control-plane) e aponta que o onboarding automático em escala é o provável gatilho para [[10.1-multi-tenant-db-compartilhado]].
- Meio-provisionamento (falha parcial) é o maior risco operacional — exigir idempotência e rollback.

## Contexto Técnico
Kit atual: `provisioning/` (schema consolidado + `setup.ts` + `update-all.ts`). Auth já é Supabase Auth SSR + RBAC (story 5.2). O gatilho de provisionamento consome o webhook de pagamento de 6.1.
