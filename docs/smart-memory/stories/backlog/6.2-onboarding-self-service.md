# Story 6.2: Onboarding self-service

> ✅ **Passo "conectar WhatsApp" desbloqueado por [[../../decisions/ADR-006-canal-whatsapp-em-escala]].** Só o passo de conectar/provisionar o WhatsApp estava atrelado ao ADR do canal; ele agora está resolvido — o canal segue **Evolution** no curto prazo, então o passo reusa o QR connect (story 5.9). O fluxo cadastro→pagamento→conta nunca esteve bloqueado.

## User Story
Como **novo cliente (corretora/agência)** quero **me cadastrar no site, pagar e ter minha conta provisionada sozinha, conectando meu WhatsApp na sequência** para **começar a usar o produto sem depender de instalação manual feita pela Cranium**.

## Objetivo
Automatizar o fluxo site → cadastro → pagamento → **conta provisionada automaticamente** → cliente conecta o próprio WhatsApp, eliminando o passo manual (kit `provisioning/`) que hoje trava a venda em escala.

## Acceptance Criteria
- [ ] AC1: Fluxo de cadastro público: visitante cria conta (e-mail + senha via Supabase Auth) e escolhe um plano vindo de [[6.1-cobranca-assinatura]].
- [ ] AC2: Após pagamento confirmado (webhook de [[6.1-cobranca-assinatura]]), a conta é **provisionada automaticamente** sem intervenção humana — o que hoje o kit `provisioning/` (`setup.ts`) faz manualmente vira gatilho automático.
- [ ] AC3: A conta provisionada nasce com schema/config prontos e o cliente é redirecionado para um **wizard de primeiros passos** (detalhado em [[9.3-wizard-setup-in-app]]) — no mínimo: conectar WhatsApp, configurar agente.
- [ ] AC4: O passo "conectar WhatsApp" reusa o QR connect (story 5.9), pois [[../../decisions/ADR-006-canal-whatsapp-em-escala]] manteve a **Evolution** no curto prazo. Quando o gatilho de migração do ADR-006 disparar (Cloud API), este passo vira o fluxo de verificação/onboarding oficial da Meta.
- [ ] AC5: Provisionamento é **idempotente e observável**: se falhar no meio, é retryável e não deixa conta meio-criada; erros notificam o super-admin ([[6.3-painel-super-admin]]).
- [ ] AC6: Tempo total do "pagou → conta pronta para configurar" é medido e registrado (meta mensurável: < 5 min sem intervenção humana no caminho feliz).

## Escopo

**IN:**
- Cadastro público + seleção de plano + integração com o checkout de [[6.1-cobranca-assinatura]].
- Gatilho automático de provisionamento pós-pagamento (evoluir o kit `provisioning/setup.ts` para rodar sob demanda/serverless).
- Redirecionamento para o wizard inicial.

**OUT:**
- O conteúdo detalhado do wizard in-app — fica em [[9.3-wizard-setup-in-app]].
- A decisão do canal WhatsApp (resolvida em [[../../decisions/ADR-006-canal-whatsapp-em-escala]]).
- Cobrança/planos em si — fica em [[6.1-cobranca-assinatura]].

## Dependências
- **AC4 (conectar WhatsApp) desbloqueado por [[../../decisions/ADR-006-canal-whatsapp-em-escala]]** (Evolution + QR connect no curto prazo).
- Depende de [[6.1-cobranca-assinatura]] (checkout + webhook de pagamento) para o gatilho de provisionamento.
- O provisionamento é orquestrado pelo control-plane definido em [[../../decisions/ADR-008-plano-de-controle-central]].
- Relaciona com [[6.3-painel-super-admin]] (visibilidade das contas provisionadas) e [[9.3-wizard-setup-in-app]].

## Notas de risco
- **Provisionamento automático no modelo "deploy por cliente"** é o ponto mais complexo: hoje cada cliente é 1 Supabase + 1 Vercel. Provisionar isso sozinho a cada venda pode exigir automação de infra (criar projeto Supabase + deploy Vercel via API) OU antecipar parte da Fase 5 (banco compartilhado `org_id`). O [[../../decisions/ADR-008-plano-de-controle-central]] define quem orquestra isso (o control-plane) e aponta que o onboarding automático em escala é o provável gatilho para [[10.1-multi-tenant-db-compartilhado]].
- Meio-provisionamento (falha parcial) é o maior risco operacional — exigir idempotência e rollback.

## Contexto Técnico
Kit atual: `provisioning/` (schema consolidado + `setup.ts` + `update-all.ts`). Auth já é Supabase Auth SSR + RBAC (story 5.2). O gatilho de provisionamento consome o webhook de pagamento de 6.1.
