# Story 8.1: LGPD

## User Story
Como **dono do SaaS que trata dados pessoais de leads** quero **estar em conformidade com a LGPD (política de privacidade, termos, DPA, consentimento e direito de exclusão)** para **operar legalmente no Brasil e não me expor a sanções nem perder a confiança dos clientes**.

## Objetivo
Colocar o produto em conformidade com a LGPD: documentos legais, base legal de tratamento, consentimento e mecanismo de exclusão de dados de titular.

## Acceptance Criteria
- [ ] AC1: **Política de privacidade** e **termos de uso** publicados e vinculados no cadastro/portal; usuário aceita explicitamente no onboarding ([[6.2-onboarding-self-service]]).
- [ ] AC2: **DPA (acordo de tratamento de dados)** disponível para o cliente (que é controlador dos dados dos leads dele; a Cranium é operadora) — relaciona [[8.2-contratos-sla-cnpj-nf]].
- [ ] AC3: **Consentimento/base legal** registrado para o tratamento dos dados de leads; a mensagem de abertura/atendimento respeita a base legal aplicável.
- [ ] AC4: **Direito de exclusão**: existe um fluxo que apaga (ou anonimiza) os dados pessoais de um lead/titular sob solicitação, propagando para todas as tabelas onde há PII (leads, messages, conversations etc.) e para backups conforme política.
- [ ] AC5: **Minimização e retenção**: política de retenção definida; logs/telemetria não guardam PII desnecessária (relaciona [[7.3-observabilidade]]).
- [ ] AC6: Registro das operações de exclusão/solicitações de titular para fins de prestação de contas (accountability).

## Escopo

**IN:**
- Documentos legais (privacidade, termos, DPA) integrados ao produto.
- Mecanismo técnico de exclusão/anonimização de dados de titular.
- Política de retenção e minimização (incluindo scrubbing de logs).

**OUT:**
- Contratos comerciais/SLA da mensalidade e nota fiscal — [[8.2-contratos-sla-cnpj-nf]] (o DPA é a fronteira compartilhada).
- Compliance específico dos termos do WhatsApp — [[8.3-termos-whatsapp]].

## Dependências
- Fronteira com [[8.2-contratos-sla-cnpj-nf]] (DPA/contratos) e [[7.3-observabilidade]] (scrubbing de PII).
- Exclusão de dados toca o schema — coordenar com crm-data.

## Notas de risco
- **Exclusão de titular no modelo "deploy por cliente"** precisa varrer o banco de cada cliente; no futuro banco compartilhado ([[10.1-multi-tenant-db-compartilhado]]), a exclusão precisa respeitar `org_id`. Projetar o fluxo para os dois cenários.
- Requer validação jurídica externa (documentos legais). O time técnico entrega o mecanismo; a redação legal pode precisar de advogado — sinalizar ao lead.

## Contexto Técnico
PII de leads espalhada por `leads`, `messages`, `group_messages`, e potencialmente e-mail. O direito de exclusão exige mapear todos os pontos de PII. Onboarding (6.2) é onde o aceite de termos entra.
